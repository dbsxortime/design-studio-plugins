# 에셋 플레이북

> 이 문서는 /brand 미팅 4(적용/에셋)를 진행하는 AI가 읽고 실행한다. 로고와 그래픽을 실제 채널(파비콘·앱 아이콘·OG 이미지·소셜 커버·이메일)에 얹는 단계다. 각 채널은 크기·크롭·렌더링 환경이 달라, 로고를 그대로 쑤셔 넣으면 반드시 깨진다. 채널별 규율을 지켜라.

`og-template.html`은 **전체 HTML 문서가 아니라 콘텐츠 프래그먼트**다 — og-render가 이 조각을 1200×630 캡처 박스(`id="cap-og"`)로 감싼다. 템플릿 안에는 플레이스홀더 `{{TITLE}}`(글 제목)과 `{{SLOGAN}}`(슬로건)만 두고, 슬로건 기본값은 brand.json의 `applications.og.slogan`에 기록한다. 웹매니페스트는 `.design/brand/icons/manifest.webmanifest`에 두어 같은 폴더의 아이콘을 상대 참조하게 한다.

---

## 1. 파비콘 축소 단순화 4원칙

로고를 16px 파비콘으로 그대로 줄이면 잉크 뭉치가 된다. 파비콘은 로고의 축소가 아니라 로고의 **재작도**다. 4원칙을 지켜라.
1. **획 수 감소**: 로고의 여러 획을 대표 실루엣 하나로 줄인다. 왜 — 16px에는 물리적으로 몇 개의 획밖에 안 들어간다.
2. **디테일 제거**: 내부 장식·미세 곡선·그러데이션을 버린다. 왜 — 작은 크기에서 디테일은 노이즈로만 보인다.
3. **광학 보정**: 축소 후 획이 얇아 보이면 굵히고, 답답하면 여백을 키운다. 왜 — 큰 크기에서 맞춘 비율은 작은 크기에서 다르게 보인다.
4. **16px 가독 테스트**: 실제 16px로 렌더해 형태가 인식되는지 눈으로 확인한다. 왜 — 코드상 크기가 아니라 실제 픽셀에서 살아남아야 파비콘이다.

## 2. maskable 아이콘 안전영역

안드로이드 홈 화면 아이콘은 OS가 원·둥근사각 등으로 마스킹한다. 로고는 **중앙 80% 안전영역** 안에 들어와야 한다(바깥 10% 마진은 마스크에 잘려 나감).

```
┌─────────────────┐  ← 512×512 캔버스
│   ┌─────────┐   │
│   │         │   │  ← 안전영역 = 중앙 80% (마진 10%씩)
│   │  로고    │   │     로고·핵심 요소는 이 안에만
│   │         │   │
│   └─────────┘   │
└─────────────────┘  ← 바깥 10%는 OS 마스크에 잘림
```

왜 — 기기·런처마다 마스크 모양이 달라, 안전영역을 벗어난 요소는 어떤 기기에서 잘릴지 예측할 수 없다. 중앙 80%는 모든 마스크가 보존하는 공통 영역이다.

---

## 3. OG 조판 3종

동적 OG 이미지(1200×630)의 조판을 아래 3종 중 하나로 정하고 og-template.html에 CSS로 굳힌다. 각 수치는 그대로 템플릿에 넣어라.
- **중앙형**: 제목을 정중앙, 로고를 상단 또는 하단 중앙. 여백은 상하좌우 대칭. 가장 안전하고 균형적.
- **좌우분할형**: 왼쪽 텍스트 블록 + 오른쪽 그래픽/패턴. 텍스트는 왼쪽 55% 폭 안에서 좌측 정렬. 편집적·정보적.
- **패턴배경형**: 브랜드 패턴을 배경 전면에 깔고 그 위에 스크림(반투명 오버레이) + 텍스트. 강한 브랜드 인상.

## 조판 크래프트 규칙

모든 조판형 에셋(OG·소셜 커버·이메일 헤더)에 공통으로 적용한다. 동적으로 채워지는 제목이 틀을 깨지 않게 하는 것이 핵심이다.

1. **텍스트 대비 ≥ 4.5:1**: 본문 텍스트와 배경의 명도 대비를 4.5:1 이상 확보한다. 패턴·이미지 배경 위 텍스트에는 **스크림**(반투명 검정/흰 오버레이)을 반드시 깐다. 왜 — 대비가 부족하면 저시력 사용자는 물론 축소 썸네일에서 아무도 제목을 못 읽는다. 스크림 없이 사진 위에 흰 글자를 얹으면 밝은 부분에서 글자가 증발한다.

2. **제목 길이 단계별 폰트 스케일**: 제목 글자 수에 따라 폰트 크기를 단계로 낮춘다 — ≤16자 **88px** / ≤28자 **64px** / 초과 시 **48px** + 최대 2줄 말줄임(ellipsis). 이 로직은 og-template의 CSS에 내장한다. 왜 — 동적 OG는 제목 길이를 예측할 수 없다. 고정 크기로 두면 긴 제목이 박스를 뚫고 나가 레이아웃이 깨진다.

3. **여백 그리드**: 외곽 마진 = 짧은 변의 8%. 로고와 텍스트 사이 간격 ≥ 로고 높이의 0.5배(클리어스페이스 규정 연동). 왜 — 마진을 변 비율로 잡으면 모든 규격에서 균형이 유지된다. 로고 클리어스페이스를 지켜야 로고가 답답하게 붙지 않는다.

4. **로고 최소 크기·고정 위치**: 커버류에서 로고는 좌상단 또는 중앙 정렬 중 **조판당 하나로 고정**한다. 왜 — 위치가 이미지마다 바뀌면 채널 피드에서 브랜드가 산만해 보인다. 고정된 로고 자리가 무의식적 인지를 만든다.

## 4. 소셜 규격표

채널별 크롭 주의를 반드시 명기하라 — 안전영역을 벗어난 텍스트·로고는 특정 기기에서 잘린다.

| 채널 | 규격(px) | 안전영역·크롭 주의 |
|------|----------|--------------------|
| OG(공용) | 1200×630 | 페이스북·슬랙·카카오 썸네일은 상하 살짝 크롭될 수 있어 텍스트를 세로 중앙에 |
| X(트위터) 커버 | 1500×500 | 좌하단이 프로필 사진·이름에 가려짐 — 핵심 요소는 우측/상단 |
| 유튜브 커버 | 2048×1152 | **안전영역 1235×338만** 모든 기기(TV/데스크톱/모바일) 공통 표시. 이 밖은 크롭 |
| 링크드인 커버 | 1128×191 | 좌측이 프로필 원형에 가려짐 — 로고·텍스트는 우측으로 |
| 프로필 | 800×800 | 원형으로 마스킹됨 — 모서리에 요소 두지 말 것 |

왜 — 소셜 커버는 채널마다 UI가 특정 영역을 덮는다. 규격만 맞추고 크롭을 무시하면 로고가 프로필 사진에 가려지는 참사가 난다.

## 5. 이메일 서명 / 헤더 규약

이메일은 웹이 아니다 — 20년 전 렌더링 엔진을 쓰는 클라이언트가 아직 살아 있다. 안전 규약을 지켜라.
- **테이블 레이아웃**: flex·grid 금지, `<table>`로 조판. 왜 — 구형 아웃룩은 최신 CSS 레이아웃을 렌더하지 못해 무너진다.
- **인라인 스타일**: `<style>` 블록 대신 각 요소에 style 속성. 왜 — 다수 웹메일이 `<head>`의 style을 제거한다.
- **표준 폭 600px**: 본문 폭을 600px로 고정. 왜 — 대부분 클라이언트의 프리뷰 창 폭이며, 이보다 넓으면 가로 스크롤이 생긴다.
- **웹세이프 폰트 폴백 스택**: 브랜드 폰트 뒤에 `sans-serif`·`serif` 같은 시스템 제너릭 폰트를 폴백으로 둔다. 왜 — 메일 클라이언트는 웹폰트를 대부분 차단해 폴백이 실제 표시 폰트가 된다.
- **이미지 alt 필수·절대경로**: 모든 이미지에 alt, src는 절대 URL. 왜 — 다수 클라이언트가 이미지를 기본 차단하므로 alt가 안 보이는 로고를 대신하고, 상대경로는 메일 안에서 깨진다.
- **다크모드 색 반전 주의**: 일부 메일 클라이언트가 다크모드에서 색을 임의 반전한다. 로고 주변에 흰 패딩을 두거나 투명 배경 로고를 피해 대비를 방어하라. 왜 — 반전되면 흰 배경 전제로 만든 로고가 검은 배경에 놓여 사라진다.
